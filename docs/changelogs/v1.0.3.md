# Changelog v1.0.3

All notable changes in this release.

## [1.0.3] - 2026-03-15

### Internal Improvements

---

#### DI-First Transport Configuration

All five transport packages previously stored their connection settings in `internal static` classes
(`NatsStatics`, `AzureServiceBusStatic`, `KafkaStatics`, `RabbitMqStatics`, `SqsStatics`).
These have been replaced with properly registered singleton services, matching the same pattern
already used by `IMapperConfiguration`.

| Transport | New Interface |
|-----------|--------------|
| NATS | `INatsConfiguration` |
| Azure Service Bus | `IAzureServiceBusConfiguration` |
| Kafka | `IKafkaConfiguration` |
| RabbitMQ | `IRabbitMqConfiguration` |
| AWS SQS | `ISqsConfiguration` |

Each configuration interface is registered as a singleton in the corresponding `AddNats()` /
`AddAzureServiceBus()` / `AddKafka()` / `AddRabbitMq()` / `AddSqs()` call and injected via
constructor into all servers and clients. All `Statics/` folders are removed.

#### Subject / Queue Name Resolution Moved into Configuration

Each transport previously had a standalone internal `Extensions.cs` file containing helper
methods that computed topic or queue names (e.g., `GetNatsSubject(this Type, INatsConfiguration)`).
These helpers have been absorbed into the configuration interfaces themselves:

| Interface | Methods |
|-----------|---------|
| `INatsConfiguration` | `GetSubject(Type)` |
| `IAzureServiceBusConfiguration` | `GetRequestQueue(Type)`, `GetReplyQueue(Type)` |
| `IKafkaConfiguration` | `GetRequestTopic(Type)` |
| `ISqsConfiguration` | `GetQueueName(Type)` |

All `Extensions.cs` helper files in transport projects are removed. Callers now invoke the
configuration object directly rather than passing it as an argument to a static helper.

#### EF Core Exception Accuracy Fix

`EntityFrameworkExtensions.AddEntityFrameworkCore()` was throwing
`DistributedMapException.AddProfilesFromAssemblyContaining` when entity configurations
were missing — a misleading exception name pointing at the wrong setup step.
It now throws `FxMapEntityFrameworkException.EntityConfigsMustNotBeEmpty` with the correct
message pointing to `AddEntitiesFromAssemblyContaining<TAssembly>`.

#### Core: Static Configuration → DI Singleton

`FxMapStatics` (the last remaining static configuration holder in the core library) was
replaced: `GetProfileConfig` and `GetEntityConfig` delegates are now registered as
`AddSingleton` in `AddFxMap()` and resolved from DI wherever needed.

### No Breaking Changes

All changes are **internal**. No public API surface, public interface, or public method signature
was altered. Consuming projects require no code changes when upgrading from 1.0.2.
