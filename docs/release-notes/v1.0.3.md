# Release Notes v1.0.3

## Overview

FxMap **1.0.3** is an internal quality improvement release. There are **no breaking changes** —
all public APIs remain identical to 1.0.2. The work focuses on eliminating static state across
the framework and ensuring all configuration flows through the standard .NET DI container.

---

## What Changed

### Transport Configuration: Statics → Singleton Services

In previous releases, each transport library (NATS, Azure Service Bus, Kafka, RabbitMQ, AWS SQS)
stored its connection settings in `internal static` classes. This caused two problems:

1. **Hidden global state** — impossible to safely run multiple transport configurations in the
   same process (e.g., tests, multi-tenant scenarios)
2. **Inconsistency** — the core `IMapperConfiguration` was already a proper DI singleton;
   transports were the odd ones out

Each transport now registers a configuration singleton at startup, matching the established pattern:

```csharp
// Before (internal, hidden static)
NatsStatics.TopicPrefix = "my-app";

// After (registered singleton, resolved by DI)
services.AddSingleton<INatsConfiguration>(new NatsConfiguration(topicPrefix));
```

All servers and clients receive their configuration through constructor injection — no static access anywhere.

### Topic / Queue Name Logic: Helpers → Configuration Methods

Subject and queue name computation previously lived in standalone `Extensions.cs` files, requiring
callers to pass the configuration as an argument:

```csharp
// Before
typeof(TDistributedKey).GetNatsSubject(_natsConfiguration)
typeof(TDistributedKey).GetAzureServiceBusRequestQueue(_azureServiceBusConfiguration)
```

This logic now belongs to the configuration object itself, which already holds the prefix:

```csharp
// After
_natsConfiguration.GetSubject(typeof(TDistributedKey))
_azureServiceBusConfiguration.GetRequestQueue(typeof(TDistributedKey))
```

Each configuration interface exposes the routing method(s) relevant to its transport:

| Interface | New Methods |
|-----------|------------|
| `INatsConfiguration` | `GetSubject(Type)` |
| `IAzureServiceBusConfiguration` | `GetRequestQueue(Type)`, `GetReplyQueue(Type)` |
| `IKafkaConfiguration` | `GetRequestTopic(Type)` |
| `ISqsConfiguration` | `GetQueueName(Type)` |

### EF Core: Corrected Guard Exception

`AddEntityFrameworkCore()` validates that `AddEntitiesFromAssemblyContaining<T>()` was called
first. It was throwing `DistributedMapException.AddProfilesFromAssemblyContaining` — a name
that suggests the wrong setup step. The exception is now
`FxMapEntityFrameworkException.EntityConfigsMustNotBeEmpty` with a clear message:

```
You have to call `AddEntitiesFromAssemblyContaining<TAssembly>` inside `AddFxMap(...)`
before calling `AddEntityFrameworkCore(...)`.
```

### Core: Last Static Removed

`FxMapStatics` in the core library held `GetProfileConfig` and `GetEntityConfig` as static
properties. These are now registered as `AddSingleton<GetProfileConfig>` and
`AddSingleton<GetEntityConfig>` inside `AddFxMap()`. The `FxMapStatics` class and its file
are deleted.

## Upgrading from 1.0.2

No changes required in consuming code. Update the package version:

```bash
dotnet add package FxMap --version 1.0.3
dotnet add package FxMap.Nats --version 1.0.3
dotnet add package FxMap.Kafka --version 1.0.3
dotnet add package FxMap.RabbitMq --version 1.0.3
dotnet add package FxMap.Azure.ServiceBus --version 1.0.3
dotnet add package FxMap.Aws.Sqs --version 1.0.3
dotnet add package FxMap.EntityFrameworkCore --version 1.0.3
dotnet add package FxMap.MongoDb --version 1.0.3
dotnet add package FxMap.Grpc --version 1.0.3
dotnet add package FxMap.HotChocolate --version 1.0.3
dotnet add package FxMap.Analyzers --version 1.0.3
```
